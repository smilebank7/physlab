# Extension Points

v0.1 keeps extension points explicit and small. There is no discovery scan,
plug-in lifecycle, auto-loading, or hidden registry walk. User code imports a
module, calls the public API, and owns the side effect.

After this document, the v0.1 protocol surface is frozen. Breaking changes to
these protocols require a version bump and an Oracle review.

## Current Extension Points

### Tasks

Tasks register through `register_task(name, factory)`. A third-party package can
put the call at module top level; the task becomes available only after the user
imports that module.

```python
from physlab import list_tasks, register_task
from physlab.tasks.cartpole import CartpoleTask

register_task("my_cartpole", CartpoleTask)
assert "my_cartpole" in list_tasks()
```

There is no `entry_points` discovery, no `importlib.metadata` walk, and no
auto-import of installed packages in v0.1. The hello-task example and
`tests/integration/test_extension_freedom.py` cover the explicit import pattern.

### Backends

Backends are structural Python protocols. physlab ships MuJoCo and Mock, and a
user backend can be passed directly to `make(..., backend=instance)`.

```python
from physlab import make
from physlab.backends.mujoco import MuJoCoBackend


class UserBackend(MuJoCoBackend):
    def name(self) -> str:
        return "user-mujoco"


env = make("cartpole", backend=UserBackend(), seed=0)
_, info = env.reset(seed=0)
assert info["backend"] == "user-mujoco"
env.close()
```

The backend contract is defined by `physlab.protocols.Backend` and exercised by
the backend contract tests.

### LLM Clients

LLM clients implement `complete(...)` and `name()`. v0.1 ships the canonical
opencode client plus a mock client for cached CI and smoke tests. Users can pass
their own client object to the orchestrator.

```python
class LocalClient:
    def complete(self, prompt: str, system: str | None = None, **kwargs: object) -> str:
        del prompt, system, kwargs
        return "local response"

    def name(self) -> str:
        return "local"


assert LocalClient().name() == "local"
```

### Reward Functions

Reward functions can be hand-written or generated by an LLM. The evaluator takes
a code string, validates syntax and imports, then executes it in a subprocess
sandbox instead of in the main process.

```python
from physlab.orchestrator.reward_gen import RewardCode

reward = RewardCode(
    code="def reward_fn(obs, action, info):\n    return 0.0\n",
    signature="reward_fn(obs, action, info)",
    hash="example",
    status="ok",
)
assert reward.status == "ok"
```

## Future Roadmap

These are documented so users know where the boundaries are, but they are not
implemented in v0.1:

- Auto-discovery through package `entry_points`: v0.2, after an explicit spec.
- Plug-in lifecycle hooks such as load, unload, and sandboxing: v0.2.
- `PolicyServer` implementations for VLA serving: v0.2.
- Procedural environment generators: v0.2.
- Multi-LLM adapter layers: v0.2.
- Drake or Taichi backends: v0.2.

Do not describe these as available in v0.1, and do not add implementation files
for them without changing the versioned scope.

## Design Principles

- Protocol-based: current boundaries use structural typing through
  `typing.Protocol`.
- Rule of Three before generalizing: do not extract a new abstraction until at
  least three concrete uses exist.
- No machinery beyond explicit function calls in v0.1: user code imports
  modules and calls APIs; discovery, lifecycle, and sandboxed plug-in loading
  belong to v0.2.
- Any addition must be justified by at least three concrete uses and reviewed by
  the Oracle before it changes the public surface.
